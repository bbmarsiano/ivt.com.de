# Directus Schema Management

This directory contains Directus schema snapshots and documentation for managing the CMS schema.

## Overview

The Directus schema is managed programmatically using scripts that create collections and fields via the Directus API. This ensures consistent schema across environments and allows version control of the schema definition.

## Schema Snapshots

We maintain **two** schema snapshots:

1. **`ivt-schema.snapshot.json`** - Spec-based snapshot (original)
   - Created from initial requirements specification
   - May include collections/fields not yet implemented in code
   - Use for reference or when implementing new features

2. **`ivt-schema.from-code.json`** - Code-based snapshot (source of truth)
   - Generated from actual TypeScript interfaces in `lib/types/content.ts`
   - Reflects exactly what the code expects
   - **This is the authoritative schema** - Directus should match this

3. **`current.directus.json`** - Current Directus state (exported)
   - Exported from running Directus instance
   - Generated by `scripts/directus-export-current-schema.js`
   - Used for comparison/diffing

See `MODEL_CONTRACT.md` for detailed field-by-field documentation of code models.

## Quick Start

### 1. Start Directus

```bash
docker compose -f docker/docker-compose.directus.yml up -d
```

### 2. Set Up Authentication

**Recommended: Create Static Admin Token**

1. Open Directus UI: http://localhost:8055
2. Login with admin credentials
3. Go to **Settings → Access Tokens**
4. Click **Create Token**
5. Set **Name**: `Scripts Admin Token`
6. Set **Role**: `Administrator`
7. Set **Expiration**: `Never`
8. Copy the token

**Store token:**

Option A: Add to `.env`:
```env
DIRECTUS_TOKEN=your-token-here
```

Option B: Create `.env.directus` (gitignored):
```bash
cp .env.directus.example .env.directus
# Edit .env.directus with your token
```

### 3. Check Status

```bash
./scripts/directus-status.sh
```

### 3. Export Current Schema (Optional)

Before making changes, export the current Directus schema:

```bash
node scripts/directus-export-current-schema.js
```

This creates `directus/snapshots/current.directus.json` with the current state.

### 4. Compare Schema

Compare current Directus schema with code expectations:

```bash
node scripts/directus-diff-schema.js
```

This shows differences between:
- Current Directus schema (`current.directus.json`)
- Code-based snapshot (`ivt-schema.from-code.json`)

**Exit codes:**
- `0` - Schemas match (no differences)
- `1` - Differences found

### 5. Apply Schema (Safe Workflow)

**⚠️ Always follow this workflow before applying schema changes:**

1. **Export current state:**
   ```bash
   node scripts/directus-export-current-schema.js
   ```

2. **Compare with code:**
   ```bash
   node scripts/directus-diff-schema.js
   ```

3. **Review differences** - Understand what will change

4. **Apply schema** (only if you're ready):
   ```bash
   ./scripts/directus-apply-schema.sh
   ```

   **Or reset and apply** (if schema needs to be reset to match code):
   ```bash
   DIRECTUS_RESET=1 ./scripts/directus-apply-schema.sh
   ```

5. **Verify** (optional):
   ```bash
   node scripts/directus-export-current-schema.js
   node scripts/directus-diff-schema.js
   ```

This ensures you never accidentally break existing data or configurations.

### Reset and Apply (Full Reset Workflow)

If Directus schema doesn't match code and needs a complete reset:

1. **Export current state** (backup):
   ```bash
   node scripts/directus-export-current-schema.js
   ```

2. **Compare to see differences:**
   ```bash
   node scripts/directus-diff-schema.js
   ```

3. **Reset CMS collections and apply code schema:**
   ```bash
   DIRECTUS_RESET=1 ./scripts/directus-apply-schema.sh
   ```

   This will:
   - Delete only CMS collections defined in code snapshot (projects, events, news, testimonials, etc.)
   - **Preserve** Prisma tables (ProjectApplication, _prisma_migrations)
   - **Preserve** any collections not in the code snapshot
   - Apply the code-based schema from `ivt-schema.from-code.json`

4. **Verify:**
   ```bash
   node scripts/directus-export-current-schema.js
   node scripts/directus-diff-schema.js
   ```
   Should show 0 differences (or only expected system fields).

## Scripts

### `scripts/directus-status.sh`

Checks if Directus is running and accessible.

**Usage:**
```bash
./scripts/directus-status.sh
```

### `scripts/directus-export-current-schema.js`

Exports the current Directus schema to a snapshot file.

**Usage:**
```bash
node scripts/directus-export-current-schema.js
```

**Output:** `directus/snapshots/current.directus.json`

**What it does:**
- Authenticates using `DIRECTUS_TOKEN` (if set) or email/password
- Uses REST system endpoints for schema metadata:
  - `GET /collections` - Lists all collections (schema level, no item querying)
  - `GET /fields` - Fetches all fields at once (schema level)
  - `GET /fields?filter[collection][_eq]={name}` - Fallback: fetches fields per collection (global endpoint)
  - `GET /relations` - Fetches all relations at once (schema level)
- Exports collection metadata, fields, and relations
- Includes all user collections (CMS collections + Prisma tables like `ProjectApplication`, `_prisma_migrations`)
- Excludes only `directus_*` system collections
- Saves to JSON file for comparison

**Note**: Uses REST schema endpoints only - **never queries `/items/*` or uses GraphQL**. Uses global `/fields` endpoints with filters, **never `/fields/{collection}`** which can cause "Queried in root" permission errors.

### `scripts/directus-diff-schema.js`

Compares current Directus schema with code-based snapshot.

**Usage:**
```bash
node scripts/directus-diff-schema.js
```

**What it does:**
- Loads `current.directus.json` (exported schema)
- Loads `ivt-schema.from-code.json` (code-based snapshot)
- Compares collections, fields, types, required flags, unique constraints, enums
- Prints human-readable diff
- Exits with code 0 if identical, 1 if differences exist

**Example output:**
```
Comparing Directus schema with code snapshot...

✗ Found 3 difference(s):

Missing Fields:
  - Field "projects.coordinator" exists in code but not in Directus
  - Field "projects.documents" exists in code but not in Directus

Type Mismatches:
  - Field "projects.images": type mismatch (Directus: files, Code: json)
```

### `scripts/directus-hard-reset-meta.js`

Hard reset broken meta-only CMS collections. Removes collection and field metadata that has no physical database tables.

**Usage:**
```bash
node scripts/directus-hard-reset-meta.js
```

**What it does:**
- Authenticates using `DIRECTUS_TOKEN` (if set) or email/password
- For each CMS collection in code snapshot (projects, events, news, testimonials):
  - Lists fields using `GET /fields?filter[collection][_eq]={name}` (global endpoint)
  - Deletes field metadata via `DELETE /fields/{collection}/{field}` (best-effort, handles 403/404 gracefully)
  - Deletes collection metadata via `DELETE /collections/{name}`
  - Treats 204 as success, 404 as "already deleted"
- Verifies collections are removed from `/collections` list
- **Preserves** Prisma tables (ProjectApplication, _prisma_migrations)
- Only targets collections defined in `ivt-schema.from-code.json`

**When to use:**
- Collections exist in Directus metadata but have no physical database tables
- Collections show 0 fields in export
- `curl "/fields?filter[collection][_eq]=projects"` returns empty array
- Need to completely remove broken metadata before recreating

**Note**: Uses global `/fields` endpoint with filters - **never `/fields/{collection}`** which can cause "Queried in root" errors.

### `scripts/directus-reset-cms-collections.js`

Resets CMS collections by deleting only collections defined in the code snapshot.

**Usage:**
```bash
node scripts/directus-reset-cms-collections.js
```

**What it does:**
- Authenticates using `DIRECTUS_TOKEN` (if set) or email/password
- Lists all collections (excluding `directus_*` system collections)
- Deletes **only** collections that are in `ivt-schema.from-code.json`
- **Preserves** Prisma tables (ProjectApplication, _prisma_migrations)
- **Preserves** any collections not in the code snapshot
- Prints which collections were deleted

**Safety:**
- Never deletes collections not in the code snapshot
- Never deletes Prisma-managed tables
- Only resets CMS collections (projects, events, news, testimonials, etc.)

**Note:** For broken meta-only collections, use `directus-hard-reset-meta.js` instead.

### `scripts/generate-directus-snapshot-from-code.js`

Generates a Directus CLI-compatible snapshot YAML by merging a canonical Directus snapshot with the code-derived JSON schema.

**Usage:**
```bash
node scripts/generate-directus-snapshot-from-code.js
```

**What it does:**
- **Input A**: `directus/snapshots/canonical.yaml` - Real Directus-generated snapshot (template)
- **Input B**: `directus/snapshots/ivt-schema.from-code.json` - Code-derived schema
- **Output**: `directus/snapshots/ivt-directus.snapshot.yaml` - Merged snapshot in Directus-native format

**Merge strategy (canonical-merge approach):**
- Starts with a canonical snapshot generated by Directus CLI (ensures correct structure)
- Removes stale entries for CMS collections (`projects`, `events`, `news`, `testimonials`)
- Adds/updates collections and fields from code schema
- **Skips "id" field** - Directus has built-in integer primary key, cannot be overridden
- **Adds "uuid" field** - Public identifier (type: uuid, unique, required) for each CMS collection
- **Ensures slug is unique and required** - Where present, slug is set as unique and required
- Preserves all other canonical snapshot parts (system settings, roles, etc.)
- Ensures exact Directus v11.14.1 snapshot format compatibility

**Why canonical-merge?**
- Directus CLI expects exact snapshot structure (version, vendor, collections, fields, relations, systemFields)
- Manually constructing YAML can miss required fields or use wrong structure
- Using Directus-generated canonical ensures format correctness
- Only CMS collections are modified; everything else stays intact

**Requirements:**
- `canonical.yaml` must exist (generated by `directus-schema-apply-container.sh` first)
- `js-yaml` package installed (`npm install js-yaml`)

### `scripts/directus-schema-apply-container.sh`

Applies Directus schema snapshot using Directus CLI inside the Docker container.

**Usage:**
```bash
./scripts/directus-schema-apply-container.sh
```

**What it does:**
1. **Step 1**: Generates canonical snapshot from Directus (`npx directus schema snapshot`)
   - Creates template with correct Directus snapshot structure
   - Copies to host as `directus/snapshots/canonical.yaml`
2. **Step 2**: Generates merged snapshot (`generate-directus-snapshot-from-code.js`)
   - Merges canonical snapshot with code schema
   - Outputs `directus/snapshots/ivt-directus.snapshot.yaml`
3. **Step 3**: Copies merged snapshot to container
4. **Step 4**: Applies schema via CLI: `npx directus schema apply --yes`
5. **Step 5**: Verifies by creating a snapshot of current state
   - Copies verification snapshot back to host as `after-apply.yaml`

**Requirements:**
- Container `ivt_directus` must be running
- Node.js installed on host (for snapshot generation)
- `js-yaml` package installed (`npm install js-yaml`)
- Directus CLI available inside container (included in Directus image)

**Advantages over API-based approach:**
- Uses official Directus CLI (more reliable)
- Avoids permission issues (CLI runs with container's Directus context)
- Creates physical database tables correctly
- Better error messages from CLI
- Canonical-merge ensures exact format compatibility with Directus v11.14.1

### `scripts/directus-fix-permissions.js`

Fixes Directus permissions for CMS collections to allow access via static admin token.

**Usage:**
```bash
node scripts/directus-fix-permissions.js
```

**What it does:**
- Authenticates using `DIRECTUS_TOKEN` (required)
- Gets current user's role via `GET /users/me`
- For each CMS collection (`projects`, `events`, `news`, `testimonials`):
  - Ensures permissions exist for actions: `read`, `create`, `update`, `delete`
  - Uses minimal permissive rules:
    - `fields: ["*"]` - All fields accessible
    - `permissions: {}` - No field-level restrictions
    - `validation: {}` - No validation rules
    - `presets: {}` - No presets
  - Creates new permissions if missing
  - Updates existing permissions to ensure correct rules
- Only touches permissions for the 4 CMS collections (safe and idempotent)

**Requirements:**
- `DIRECTUS_TOKEN` must be set (in `.env.directus` or environment)
- Token must have admin permissions to create/update permissions

**When to use:**
- After creating collections via schema apply
- When getting "FORBIDDEN: You don't have permission to access collection" errors
- When Directus UI shows permission errors for CMS collections

### `scripts/directus-fix-permissions.sh`

Shell wrapper for the permissions fix script.

**Usage:**
```bash
./scripts/directus-fix-permissions.sh
```

**What it does:**
- Loads environment variables from `.env.directus` (if present)
- Validates `DIRECTUS_TOKEN` is set
- Runs `directus-fix-permissions.js`
- Prints verification commands

### `scripts/directus-apply-schema.sh`

Applies the schema snapshot to Directus via REST API.

**Usage:**
```bash
# Apply only (creates missing collections/fields)
./scripts/directus-apply-schema.sh

# Reset CMS collections first, then apply
DIRECTUS_RESET=1 ./scripts/directus-apply-schema.sh
```

**What it does:**
- If `DIRECTUS_RESET=1`: Runs hard reset script first, then applies schema
- Otherwise: Only applies schema (creates missing collections/fields)
- Authenticates using `DIRECTUS_TOKEN` (if set) or email/password
- Creates collections if they don't exist (with UUID primary keys)
- **Verifies physical tables exist** by checking field counts after creation
- Creates fields for each collection via `POST /fields` (global endpoint with `collection` in body)
- Verifies final field count matches expected
- Creates relations to `directus_files` for file fields
- Lists all created collections

**Physical Table Creation:**
- Uses `POST /collections` with explicit schema to create physical tables
- Creates `id` field as UUID primary key immediately after collection creation
- Verifies collection has fields after creation (throws error if 0 fields)
- Verifies final field count after all fields are created
- Ensures physical database tables exist, not just metadata

**Environment Variables:**
- `DIRECTUS_URL` - Directus URL (default: `http://localhost:8055`)
- `DIRECTUS_ADMIN_EMAIL` - Admin email (or `ADMIN_EMAIL` from `.env`)
- `DIRECTUS_ADMIN_PASSWORD` - Admin password (or `ADMIN_PASSWORD` from `.env`)
- `DIRECTUS_RESET` - Set to `1` to reset CMS collections before applying (default: `0`)

## Schema Snapshots

We maintain multiple schema representations:

1. **`ivt-schema.from-code.json`** - Code-derived JSON schema (source of truth)
   - Generated from TypeScript interfaces
   - Used by API-based apply scripts
   - Used as input for CLI snapshot generator

2. **`canonical.yaml`** - Directus-generated canonical snapshot (template)
   - Generated by `npx directus schema snapshot` inside container
   - Contains exact Directus v11.14.1 snapshot structure
   - Used as template for merging code schema
   - Preserves system settings, roles, and other Directus metadata

3. **`ivt-directus.snapshot.yaml`** - Merged Directus CLI snapshot (generated)
   - Generated by merging `canonical.yaml` + `ivt-schema.from-code.json`
   - Uses canonical-merge approach for format compatibility
   - Used by `directus schema apply` command
   - Directus CLI-compatible format

4. **`current.directus.json`** - Current Directus state (exported)
   - Exported from running Directus instance via REST API
   - Used for comparison/diffing

5. **`after-apply.yaml`** - Verification snapshot (generated after apply)
   - Created after schema apply for debugging/verification
   - Shows actual state after applying merged snapshot

The schema can be applied via:
- **Directus CLI** (recommended): `./scripts/directus-schema-apply-container.sh`
- **REST API**: `./scripts/directus-apply-schema.sh`

## Primary Key and Identifiers

**Directus Built-in Primary Key:**
- Directus automatically creates an integer `id` field as the primary key for all collections
- This is a system field and should **not** be defined in snapshots
- The generator automatically skips any "id" field from the code schema

**Public Identifier (UUID):**
- Each CMS collection has a `uuid` field (type: uuid) that serves as the public identifier
- This field is:
  - Required: `true`
  - Unique: `true`
  - Auto-generated by PostgreSQL (via `gen_random_uuid()`)
- In Next.js content mapping:
  - `id` (from code) → `uuid` (in Directus)
  - `slug` remains the unique identifier for routing

**Slug for Routing:**
- `slug` field is unique and required (where present)
- Used as the main identifier for URL routing
- Example: `/projects/ai-research-hub` uses the slug

## Field Naming Convention

All content fields use explicit language suffixes:
- `*_de` - German content
- `*_en` - English content

Examples:
- `title_de`, `title_en`
- `description_de`, `description_en`
- `summary_de`, `summary_en`

## File Relations

File fields are linked to `directus_files` collection:
- **Many-to-one (M2O)**: Single file (e.g., `cover_image`, `logo`, `photo`)
- **Many-to-many (M2M)**: Multiple files (e.g., `images`, `documents`)

## Troubleshooting

### "Container not running"

**Error**: `Docker container 'ivt_directus' is not running`

**Solution**:
```bash
docker compose -f docker/docker-compose.directus.yml up -d
```

### "API not accessible"

**Error**: `Directus API is not accessible`

**Solutions**:
1. Check container logs:
   ```bash
   docker compose -f docker/docker-compose.directus.yml logs
   ```
2. Verify port 8055 is not in use
3. Check `DIRECTUS_URL` environment variable

### "403 FORBIDDEN" or "Login failed"

**Error**: `403 FORBIDDEN` or `Login failed`

**Solutions:**

1. **Use Static Admin Token (Recommended)**:
   - Create a static token in Directus UI: Settings → Access Tokens
   - Set role to `Administrator`
   - Add to `.env` or `.env.directus`:
     ```env
     DIRECTUS_TOKEN=your-static-token-here
     ```
   - Static tokens avoid permission issues and work better with REST API

2. **If using email/password**:
   - Verify admin credentials in `.env`:
     ```env
     DIRECTUS_ADMIN_EMAIL=admin@example.com
     DIRECTUS_ADMIN_PASSWORD=your-password
     ```
   - Check credentials match `docker/directus.env`
   - Ensure Directus has finished initializing (wait 30-60 seconds after starting)
   - Verify admin user has proper permissions in Directus

3. **Check authentication method**:
   - Scripts print which auth method is used
   - Ensure either `DIRECTUS_TOKEN` or `DIRECTUS_ADMIN_EMAIL`/`DIRECTUS_ADMIN_PASSWORD` is set

### "Collection already exists"

**Info**: This is normal - the script skips existing collections/fields. To recreate:
1. Delete collection in Directus UI (Settings > Data Model)
2. Re-run `./scripts/directus-apply-schema.sh`

### "Field creation failed"

**Error**: Field creation errors

**Solutions**:
1. Check if field name conflicts with reserved names
2. Verify field type is supported
3. Check Directus logs for detailed error messages

### Schema changes not reflected

**Issue**: Changes to `directus-apply-schema.js` not applied

**Solution**:
1. Delete affected collections in Directus UI
2. Re-run `./scripts/directus-apply-schema.sh`
3. Or manually update fields via Directus UI

### "FORBIDDEN: You don't have permission to access collection"

**Error**: `FORBIDDEN: You don't have permission to access collection ... Queried in root`

**Symptoms:**
- Directus UI shows FORBIDDEN when accessing CMS collections
- `curl /items/projects` returns 403 FORBIDDEN
- Collections exist in database but permissions are missing

**Solution:**
Run the permissions fix script:
```bash
./scripts/directus-fix-permissions.sh
```

This script:
- Uses `DIRECTUS_TOKEN` to authenticate
- Gets current user's role
- Creates/updates permissions for CMS collections (projects, events, news, testimonials)
- Sets minimal permissive rules for read, create, update, delete actions
- Only touches permissions for the 4 CMS collections (safe and idempotent)

**Verification:**
```bash
# Test API access
curl -s -H "Authorization: Bearer $DIRECTUS_TOKEN" \
  "http://localhost:8055/items/projects?limit=1" | jq
# Should return data (not FORBIDDEN)

# Check in Directus UI
# Open http://localhost:8055 → Content → Projects
# Should open without errors
```

## Updating Schema

### Method 1: Directus CLI (Recommended)

Uses official Directus CLI inside the container. More reliable and avoids permission issues.

**Canonical-Merge Approach:**
This method uses a "canonical-merge" strategy to ensure exact Directus snapshot format compatibility:
1. Generates a canonical snapshot from Directus (template with correct structure)
2. Merges code schema into canonical snapshot (only CMS collections modified)
3. Applies merged snapshot via CLI

This ensures the snapshot matches Directus v11.14.1's expected format exactly.

1. **Backup current state (optional):**
   ```bash
   # Export current schema
   node scripts/directus-export-current-schema.js
   
   # Or create snapshot inside container
   docker exec ivt_directus sh -c "npx directus schema snapshot /directus/snapshots/backup-$(date +%Y%m%d).yaml"
   ```

2. **Update code snapshot** (if code models changed):
   - Edit `directus/snapshots/ivt-schema.from-code.json` to match new code
   - Or update `MODEL_CONTRACT.md` first, then update snapshot

3. **Generate and apply snapshot:**
   ```bash
   ./scripts/directus-schema-apply-container.sh
   ```
   
   This will:
   - **Step 1**: Generate canonical snapshot from Directus (`canonical.yaml`)
   - **Step 2**: Merge canonical with code schema (`ivt-directus.snapshot.yaml`)
   - **Step 3**: Copy merged snapshot to container
   - **Step 4**: Apply via `npx directus schema apply --yes`
   - **Step 5**: Verify by creating a snapshot of current state (`after-apply.yaml`)

4. **Fix permissions (if needed):**
   ```bash
   ./scripts/directus-fix-permissions.sh
   ```
   
   This ensures your token has permissions to access the CMS collections.

5. **Verify:**
   ```bash
   # Check in Directus UI
   # Open http://localhost:8055 → Settings > Data Model
   # Verify collections: projects, events, news, testimonials
   # Check that fields are visible (no 403 errors)
   # Open Content → Projects - should work without FORBIDDEN errors
   
   # Check in database
   psql -d ivt_dev -c "\dt"
   # Should show: projects, events, news, testimonials
   
   # Check via API (fields)
   curl -s -H "Authorization: Bearer $DIRECTUS_TOKEN" \
     "http://localhost:8055/fields?filter[collection][_eq]=projects" | jq '.data | length'
   # Should return > 0
   
   # Check via API (items - permissions test)
   curl -s -H "Authorization: Bearer $DIRECTUS_TOKEN" \
     "http://localhost:8055/items/projects?limit=1" | jq
   # Should return data (not FORBIDDEN)
   ```

### Method 2: REST API (Alternative)

Uses custom Node.js scripts to apply schema via Directus REST API.

### Standard Update (Add New Fields/Collections)

When adding new fields or collections:

1. **Export current state:**
   ```bash
   node scripts/directus-export-current-schema.js
   ```

2. **Update code snapshot** (if code models changed):
   - Edit `directus/snapshots/ivt-schema.from-code.json` to match new code
   - Or update `MODEL_CONTRACT.md` first, then update snapshot

3. **Compare schemas:**
   ```bash
   node scripts/directus-diff-schema.js
   ```

4. **Review differences** - Understand what will change

5. **Apply schema:**
   ```bash
   # Option A: Via CLI (recommended)
   ./scripts/directus-schema-apply-container.sh
   
   # Option B: Via REST API
   ./scripts/directus-apply-schema.sh
   ```

6. **Verify:**
   - Run diff again: `node scripts/directus-diff-schema.js`
   - Check Directus UI
   - Test application functionality

### Full Reset (Schema Mismatch)

When Directus schema doesn't match code and needs complete reset:

1. **Export current state** (backup):
   ```bash
   node scripts/directus-export-current-schema.js
   ```

2. **Compare to see differences:**
   ```bash
   node scripts/directus-diff-schema.js
   ```

3. **Reset CMS collections and apply code schema:**
   ```bash
   DIRECTUS_RESET=1 ./scripts/directus-apply-schema.sh
   ```

   **What this does:**
   - Runs `directus-hard-reset-meta.js` to remove broken meta-only collections
   - Deletes collection and field metadata for CMS collections
   - Preserves Prisma tables (ProjectApplication, _prisma_migrations)
   - Applies code-based schema from `ivt-schema.from-code.json`
   - Creates physical database tables with UUID primary keys
   - Verifies physical tables exist by checking field counts

4. **Verify:**
   ```bash
   # Check fields endpoint returns data (using global endpoint with filter)
   curl -s -H "Authorization: Bearer $DIRECTUS_TOKEN" \
     "http://localhost:8055/fields?filter[collection][_eq]=projects" | jq '.data | length'
   # Should return > 0
   
   # Export and diff
   node scripts/directus-export-current-schema.js
   node scripts/directus-diff-schema.js
   ```
   Should show 0 differences (or only expected system fields).

5. **Optional: Seed data** (if you have seed scripts)

### Broken Meta-Only Collections Recovery

If collections exist in Directus metadata but have no physical database tables:

**Symptoms:**
- Collections appear in Directus UI but have 0 fields
- `curl "/fields?filter[collection][_eq]=projects"` returns empty array
- `psql \dt` shows no physical table
- Export shows collections with 0 fields

**Recovery:**

1. **Hard reset broken collections:**
   ```bash
   node scripts/directus-hard-reset-meta.js
   ```
   
   This script:
   - Deletes field metadata for broken collections
   - Deletes collection metadata
   - Verifies collections are removed
   - Only targets CMS collections from code snapshot
   - Preserves Prisma tables

2. **Recreate collections properly:**
   ```bash
   DIRECTUS_RESET=1 ./scripts/directus-apply-schema.sh
   ```
   
   This will:
   - Run hard reset first (removes broken metadata)
   - Create collections with physical tables
   - Create UUID primary keys
   - Create all fields
   - Verify physical tables exist

3. **Verify physical tables exist:**
   ```bash
   # Check fields endpoint using global endpoint with filter
   curl -s -H "Authorization: Bearer $DIRECTUS_TOKEN" \
     "http://localhost:8055/fields?filter[collection][_eq]=projects" | jq '.data | length'
   # Should return > 0 (not 0)
   
   # Or check in psql
   psql -d ivt_dev -c "\dt projects"
   # Should show the table
   ```

**Note:** The apply schema script now verifies that physical tables are created by checking field counts after collection creation. If verification fails, it throws an error.

## Best Practices

1. **Always apply schema from script** - Don't manually create collections in UI
2. **Version control schema** - Commit `directus-apply-schema.js` to git
3. **Test locally first** - Apply schema locally before production
4. **Backup before changes** - Export Directus snapshot before major schema changes
5. **Use consistent naming** - Follow `*_de`/`*_en` convention for multilingual fields

   DIRECTUS_ADMIN_EMAIL=admin@yourdomain.com
   DIRECTUS_ADMIN_PASSWORD=secure-password

2. Run schema application:
   ```bash
   ./scripts/directus-apply-schema.sh
   ```

3. Verify schema:
   - Check collections in Directus UI
   - Test API endpoints
   - Verify file relations work
